Figure 3.3: Sequence Diagram — Outbound Packet Processing

Interaction sequence from TUN read to relay, showing the allow/drop decision path.

sequenceDiagram
    participant TUN as TUN Interface
    participant VPN as QosVpnService
    participant DPP as DataPlaneProcessor
    participant RP as RawPacket.parse()
    participant AR as AppResolver
    participant DPI as DpiClassifier
    participant BS as BandwidthScheduler
    participant TB as TokenBucket
    participant RELAY as TcpRelayManager

    TUN->>VPN: read(buffer)
    VPN->>DPP: process(bytes, length, outbound)
    DPP->>RP: parse(buffer, length)
    RP-->>DPP: RawPacket or null

    alt Malformed packet
        DPP-->>VPN: Result(parsed=false)
    end

    DPP->>DPP: Check flowCache[FlowKey]

    alt Cache miss
        DPP->>AR: getUidForConnection(5-tuple)
        AR-->>DPP: UID or null
        DPP->>DPP: flowCache[flowKey] = UID
    end

    DPP->>DPI: classify(packet)
    DPI-->>DPP: TrafficCategory

    DPP->>BS: ensureBucket(appKey, priority)
    DPP->>BS: processPacket(appKey, size)
    BS->>TB: consume(bytes)
    TB-->>BS: allowed: Boolean
    BS-->>DPP: allowed: Boolean

    alt Packet ALLOWED
        DPP-->>VPN: Result(dropped=false, packet)
        VPN->>RELAY: handlePacket(parsedPacket)
        RELAY->>RELAY: protect(socket)
        RELAY->>RELAY: Forward to Internet
    else Packet DROPPED
        DPP-->>VPN: Result(dropped=true)
        Note over VPN: TCP sender detects loss,
reduces congestion window
    end

Figure 3.4: Sequence Diagram — TCP Split-Handshake Proxy

The relay mechanism that bridges VPN TUN with the real Internet via protected sockets.

sequenceDiagram
    participant App as Client App
    participant TUN as TUN Interface
    participant QoS as TcpRelayManager
    participant PC as PacketComposer
    participant SOCK as Protected Socket
    participant SRV as Remote Server

    App->>TUN: SYN packet
    TUN->>QoS: Intercepted SYN
    QoS->>SOCK: Create SocketChannel
    QoS->>QoS: protect(socket)
    QoS->>SOCK: connect(destination)
    SOCK->>SRV: Real TCP SYN
    SRV-->>SOCK: Real SYN-ACK
    QoS->>PC: Build synthetic SYN-ACK
    PC-->>TUN: Synthetic SYN-ACK
    TUN-->>App: SYN-ACK received

    App->>TUN: ACK + Data
    TUN->>QoS: Intercepted data
    QoS->>SOCK: Forward payload
    SOCK->>SRV: Real data

    SRV-->>SOCK: Response data
    QoS->>PC: Build IP/TCP packet
    PC-->>TUN: Synthetic response packet
    TUN-->>App: Response received

    Note over App,SRV: Bidirectional relay continues...

    App->>TUN: FIN
    TUN->>QoS: Intercepted FIN
    QoS->>SOCK: Close connection
    SOCK->>SRV: FIN
    QoS->>QoS: Cleanup flow resources

Figure 3.5: Activity Diagram — Packet Processing Pipeline

Decision flow from TUN interception to relay or drop.

flowchart TD
    START(["TUN Read"]) --> PARSE["RawPacket.parse(buffer)"]
    PARSE --> VALID{"Valid IP packet?"}
    VALID -->|No| DROP1["Discard silently"]
    VALID -->|Yes| CACHE{"FlowKey in cache?"}

    CACHE -->|Yes| CLASSIFY
    CACHE -->|No| UID["AppResolver.getUidForConnection()"]
    UID --> UID_OK{"UID resolved?"}
    UID_OK -->|Yes| STORE["Cache FlowKey → UID"]
    STORE --> CLASSIFY
    UID_OK -->|No| SYNTH["Assign to synthetic port bucket"]
    SYNTH --> CLASSIFY

    CLASSIFY["DpiClassifier.classify()"] --> ENSURE["BandwidthScheduler.ensureBucket()"]
    ENSURE --> CONSUME["TokenBucket.consume(packetSize)"]
    CONSUME --> ALLOWED{"Tokens available?"}

    ALLOWED -->|No| DROP2["DROP packet"]
    DROP2 --> STATS1["Increment dropped counter"]
    STATS1 --> END1(["Return to TUN read loop"])

    ALLOWED -->|Yes| MODE{"Runtime mode?"}
    MODE -->|Monitor| WRITEBACK["Write back to TUN"]
    WRITEBACK --> END2(["Return to TUN read loop"])

    MODE -->|Relay| PROTO{"Protocol?"}
    PROTO -->|TCP| TCP_R["TcpRelayManager.handlePacket()"]
    PROTO -->|UDP| UDP_R["UdpRelayManager.handlePacket()"]
    PROTO -->|Other| WRITEBACK2["Write back to TUN"]
    TCP_R --> END3(["Return to TUN read loop"])
    UDP_R --> END3
    WRITEBACK2 --> END3
