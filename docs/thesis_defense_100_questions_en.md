## CATEGORY I: KERNEL ROUTING, INTERCEPTION & VPN SERVICE LIMITATIONS (Questions 1 - 15)

**1. If the Android OS terminates (kills) the `QosVpnService` process due to low memory, how does the system recover, and what happens to active user network connections?**
*   **Answer:** Yes, professors. If the Android OS kills my service due to low RAM, the virtual TUN interface is closed immediately. The OS automatically restores the default routing table to the physical Wi-Fi or cellular interface, so the user does not experience a total network blackout. However, active proxy connections are terminated, forcing applications (like games or browsers) to reconnect. I configured the `START_STICKY` flag so the OS will automatically restart the service once memory is freed.

**2. Why does calling `protect(socket)` fail on certain MediaTek devices or custom ROMs like MIUI/HyperOS? How did you implement the fallback mechanism?**
*   **Answer:** Yes, on some devices running custom ROMs like MIUI/HyperOS or MediaTek chipsets, socket protection can fail due to system permission restrictions. To handle this, I implemented a 3-step fallback mechanism: First, I retrieve the active physical `Network` object and call `bindSocket(socket)` to force the socket routing. If that fails, I extract the raw File Descriptor to invoke `protect(fd)`. If both fail, I fall back to the standard `protect(socket)` call.

**3. Why did you configure the TUN interface's MTU to `1500` instead of using larger sizes like `9000` (Jumbo Frames) to increase userspace loopback speeds?**
*   **Answer:** Yes, I set the virtual TUN MTU to 1500 because this matches the standard MTU of physical Wi-Fi and mobile networks. If I set it higher (like 9000), packets exiting to the WAN would undergo IP fragmentation at the physical layer. This would double the number of packets sent, increasing latency (RTT) and packet loss rate.

**4. The QoS Scheduler operates in a Split Tunneling configuration. What happens if a user starts another VPN application (e.g., 1.1.1.1)? Can Android run them in parallel?**
*   **Answer:** Yes, Android only allows a single VPN application to control the `/dev/tun` interface at any given time. If another VPN starts, it revokes permissions from my app. My service receives an `onDestroy()` callback and stops scheduling traffic. This is a design limitation of the Android OS.

**5. In `QosVpnService.kt`, why did you restrict I/O execution to 8 concurrent threads (`Dispatchers.IO.limitedParallelism(8)`) instead of using default dispatchers?**
*   **Answer:** Yes, by default, `Dispatchers.IO` can expand up to 64 threads. On mobile devices with limited CPU resources, having too many concurrent threads causes thread contention and context-switching overhead. Locking the pool to 8 parallel threads ensures stable I/O performance and keeps CPU usage low.

**6. Why did you choose a buffered packet channel capacity of `4096`? What is the mathematical justification?**
*   **Answer:** Yes, I calculated the channel capacity of 4096 based on high-speed network traffic (~100 Mbps). At an average packet size of 1500 bytes, 100 Mbps equals about 8000 packets per second. A buffer size of 4096 allows the system to hold packets for up to 500ms of temporary I/O delay without overflowing and causing packet loss.

**7. Socket protection relies on binding sockets to physical interfaces. If the device switches from Wi-Fi to cellular LTE, how do you handle the invalidation of the bound interface?**
*   **Answer:** Yes, when the Wi-Fi connection drops, its network object is destroyed. To handle this, my app registers a `NetworkCallback`. When a network transition to LTE occurs, a background job automatically queries the new LTE network object and calls `bindSocket()` again on all active proxy sockets, allowing traffic to continue seamlessly.

**8. How does the system exclude its own telemetry and sync traffic from the virtual TUN interface?**
*   **Answer:** Yes, when constructing the VPN, I call `builder.addDisallowedApplication(packageName)` and pass my app's package name. The Android system then routes all network packets generated by my app directly to the physical interface, bypassing the TUN to avoid routing loops.

**9. If the TUN interface experiences an unexpected I/O error due to OS permission changes, how does the system recover?**
*   **Answer:** Yes, the packet reader loop is wrapped in a `try-catch` block. If an I/O error (like an `IOException`) occurs, the catch block detects it, calls `close()` to release the TUN interface, resets the service state, and displays a user notification.

**10. Why does the project deploy a `PacketPool` instead of allocating a new `ByteArray(32767)` for every single packet?**
*   **Answer:** Yes, allocating a new byte array for thousands of packets per second triggers Java Garbage Collection (GC) thrashing. This causes CPU spikes and makes the app lag. `PacketPool` recycles byte arrays using an `ArrayBlockingQueue(2048)`, maintaining a stable memory profile and low CPU usage.

**11. MTU 1500 of TUN combined with TCP MSS Clamping 1460. If the packet is UDP, is there an equivalent mechanism to prevent fragmentation?**
*   **Answer:** Yes, UDP is a connectionless protocol and does not have an MSS field in its header. For UDP packets larger than 1500 bytes, the Android system handles IP fragmentation at the network layer. However, mobile UDP traffic (like VoIP and gaming) usually consists of very small packets (under 500 bytes), so fragmentation is rare.

**12. Why did you choose Kotlin as the primary language for your Android app instead of Java or C++?**
*   **Answer:** Yes, professors. I chose Kotlin because it is the modern language officially recommended by Google for Android development. Kotlin helps me write concise code and prevents null pointer errors (`NullPointerException`) via its built-in Null Safety. Compared to C++ (via JNI), developing in Kotlin is faster and easier to debug, while still maintaining high performance and seamless compatibility with native Java libraries.

**13. How does `healthMonitorJob` distinguish between physical network congestion (WAN side) and throttling enforced by your Token Bucket?**
*   **Answer:** Yes, the `healthMonitorJob` pings Google's DNS (`8.8.8.8`) using a socket protected by `protect()`. This socket bypasses the TUN interface entirely and is not throttled. Therefore, if the ping latency spikes, it indicates genuine WAN-side network congestion rather than scheduler throttling.

**14. Why does the project not support IPv6 routing in the TUN interface configuration (`addRoute` is restricted to `0.0.0.0/0`)?**
*   **Answer:** Yes, supporting IPv6 would require writing separate parsing headers and maintaining additional cache tables, which doubles the CPU overhead on the device. To optimize performance, I bypass IPv6 (routing it directly to the physical interface) and focus on IPv4, which is the most common protocol.

**15. How does the system detect that a hotspot client has disconnected to release bandwidth scheduler allocations?**
*   **Answer:** Yes, the `DeviceRegistry` class scans the system ARP table every 10 seconds. If a client's IP is missing or marked inactive (`0x0`), it calls a cleanup function to remove the device from the scheduler, immediately reclaiming memory.

## CATEGORY II: USERSPACE PROTOCOL RELAY & SPLIT-HANDSHAKE VULNERABILITIES (Questions 16 - 25)

**16. Why must you deploy a Split-Handshake Proxy instead of Raw IP Forwarding?**
*   **Answer:** Yes, raw IP forwarding requires configuring NAT at the kernel level, which Android blocks for security reasons on unrooted devices. A Split-Handshake Proxy runs in userspace, allowing us to manage traffic flows and apply TCP Backpressure (by adjusting TCP window sizes) without needing root access.

**17. If a client transmits a packet containing the `RST` flag during the Split-Handshake phase, how does the proxy clean up resources?**
*   **Answer:** Yes, upon receiving a `RST` packet from the client, the proxy immediately closes the remote socket connection to the target server to reclaim the file descriptor and removes the flow from `flowCache` to prevent socket leaks.

**18. How does the proxy calculate the Sequence Number offset when the client sends data before the remote server completes the handshake?**
*   **Answer:** Yes, if the client sends data early, the proxy temporarily blocks reading from the client. It waits until it receives the `SYN-ACK` packet from the remote server to get the server's initial sequence number (ISN). Once it calculates the offset, the proxy resumes reading and translates sequence numbers accordingly.

**19. Why did you choose Java NIO Selector (Non-blocking I/O) in TcpRelayManager.kt instead of standard sockets combined with Kotlin Coroutines for each connection?**
*   **Answer:** Yes, professors. If we use traditional blocking sockets, each connection requires its own thread. When the phone runs hundreds of concurrent connections, the system gets overloaded due to thread context-switching and memory overhead. Java NIO Selector uses a Single-Thread Multiplexing mechanism, where a single selector thread monitors events across hundreds of sockets. It only processes a socket when data is actually ready, saving CPU and RAM.

**20. In Java NIO Selector (`TcpRelayManager.kt`), what happens if a physical socket's write buffer becomes full? Does the selector thread freeze?**
*   **Answer:** Yes, the selector does not freeze because Java NIO is non-blocking. If a write buffer becomes full, the write event (`OP_WRITE`) is temporarily disabled, and the data is queued in a local buffer. The thread continues to process other active connections. Once the buffer is cleared, the system re-registers `OP_WRITE` to flush the queue.

**21. Why must IP and TCP checksums be recalculated on every packet? Can we disable checksum calculations on emulators?**
*   **Answer:** Yes, even on emulators, outgoing packets are validated by the host machine's physical network card and routers. If a checksum is incorrect by even 1 bit, the packet is discarded, which breaks the network connection completely. Therefore, checksum recalculation is mandatory.

**22. How does the proxy handle TCP Sequence Number Overflow during long-duration high-volume file transfers?**
*   **Answer:** Yes, TCP sequence numbers are 32-bit integers that wrap around to 0 when they overflow. To handle this, my proxy calculates the sequence number offset using Long (64-bit) variables, then casts the result to a 32-bit unsigned integer when writing the TCP header, ensuring accurate tracking.

**23. In `UdpRelayManager.kt`, why is the UDP buffer size set to `65507` bytes instead of the standard MTU of `1500`?**
*   **Answer:** Yes, while the physical MTU is 1500 bytes, the UDP protocol allows logical packets up to 65507 bytes by fragmenting them at the IP layer. If I set the buffer to 1500, larger UDP packets would be truncated, causing application errors.

**24. What happens if the remote server sends a `FIN` flag to close the connection, but the client continues writing data to the proxy?**
*   **Answer:** Yes, the proxy switches the connection to a "Half-Closed" state. It continues to relay data from the client to the server but stops reading from the server. Once the client sends its own `FIN` packet, the connection is fully closed.

**25. Why does your TCP Backpressure mechanism control throughput without causing packet drops?**
*   **Answer:** Yes, professors. When I want to limit an app's bandwidth, the proxy temporarily stops reading from its socket. This fills up the Android OS receive buffer. The OS then automatically sends a Zero Window size notification to the client app, pausing its transmission. This throttles the app naturally without dropping network packets or wasting data.

## CATEGORY III: QoS SCHEDULING ALGORITHMS & MATHEMATICAL MODELS (Questions 26 - 35)

**26. Why did you choose the Token Bucket algorithm over the Leaky Bucket algorithm?**
*   **Answer:** Yes, the Leaky Bucket algorithm enforces a rigid transmission rate, which ruins the user experience for interactive tasks like loading a webpage or playing a game. The Token Bucket algorithm is more flexible because it allows short traffic bursts (bursts) when tokens are available, then limits traffic to the average refill rate, keeping the app responsive.

**27. In `TokenBucket.kt`, the `tokens` variable is updated using Double precision. How do you prevent Floating-Point Rounding Errors over long runs?**
*   **Answer:** Yes, I cap the maximum token count to `burstBits`. The refill calculation uses `minOf(burstBits, tokens + rate * elapsed)`. Capping the tokens at this upper limit prevents rounding errors from accumulating over time.

**28. Why does the WFQ model in `BandwidthScheduler.kt` assign the `__host__` bucket a fixed priority weight of 2?**
*   **Answer:** Yes, the `__host__` bucket manages the network traffic of the Android OS itself. If we do not assign it a minimum weight (here set to 2, equivalent to a MEDIUM app), background system services might lose connection when high-priority apps run at full speed, which could cause the VPN service to fail.

**29. Why does the system configure the burst capacity as `burstBits = newRateBps / 10`?**
*   **Answer:** Yes, this formula limits the maximum burst capacity to exactly **100 milliseconds** of transmission at the current rate. This prevents idle applications from accumulating too many tokens and sending a huge packet burst that would overwhelm the physical network card.

**30. Does using Traffic Policing (packet drops) instead of Traffic Shaping (queues) increase the TCP Retransmission Rate? Does this waste mobile data?**
*   **Answer:** Yes, dropping packets does trigger TCP retransmissions. However, this increase only occurs briefly during the initial congestion phase before the client app's TCP congestion control reduces its sending rate. We accept this minor data overhead because traffic shaping requires huge memory buffers and causes high latency (Bufferbloat).

**31. If a UDP application spams datagrams at 10 Mbps while its QoS policy is capped at 1 Mbps, how does your policing scheduler handle this?**
*   **Answer:** Yes, since UDP does not automatically reduce its sending rate like TCP does, the client continues to push packets at full speed. The Token Bucket will quickly run out of tokens, and my scheduler will drop 90% of these UDP packets at the TUN interface, keeping the outgoing speed at 1 Mbps.

**32. How does the QoS Scheduler mitigate Bufferbloat on physical routers?**
*   **Answer:** Yes, I set the total shaped bandwidth slightly lower (about 5%) than the actual physical connection speed. This shifts the network bottleneck from the physical router to the virtual TUN interface on the phone. Because our TUN interface does not accumulate large queues, the physical router's buffer remains empty, eliminating Bufferbloat.

**33. Why is `rebalanceWithApps` invoked automatically whenever an application changes its network state (Topology Change)?**
*   **Answer:** Yes, because the WFQ algorithm calculates bandwidth shares based on the total weight of active apps. When an app starts or stops using the network, the total weight changes. Calling `rebalanceWithApps` immediately recalculates the token refill rates for the remaining active apps to avoid wasting bandwidth.

**34. How does the WFQ scheduler guarantee that LOW priority apps are not completely starved of bandwidth during high system load?**
*   **Answer:** Yes, in my WFQ configuration, LOW priority apps are assigned a minimum weight of 1 (while MEDIUM is 2 and HIGH is 4). Thus, even under maximum load from high-priority apps, LOW priority apps are guaranteed a small minimum share of the bandwidth to maintain basic connectivity.

**35. Why does `rateForClass` double the burst for HIGH-priority apps (`rate * 2`) but halve it for LOW-priority apps (`rate / 2`)?**
*   **Answer:** Yes, HIGH-priority apps (like games or VoIP) send small packets that need to go out immediately, so I double their burst capacity to prevent random drops. LOW-priority apps (like downloads) send continuous large packets, so I halve their burst capacity to avoid sending huge packet bursts that could lag other apps.

## CATEGORY IV: APP IDENTIFICATION & BINDER IPC WORKAROUND LIMITATIONS (Questions 36 - 45)

**36. Why do you invoke `getConnectionOwnerUid` synchronously on the data plane thread instead of asynchronously?**
*   **Answer:** Yes, if I called it asynchronously, packets would have to wait in RAM for the UID lookup to finish, which would increase packet latency (RTT). Since the UID lookup takes only about 1ms, I call it synchronously and cache the results to get the fastest performance.

**37. Why does calling `getConnectionOwnerUid` create a system bottleneck (Binder IPC Bottleneck)?**
*   **Answer:** Yes, because this call is a system transaction (Binder IPC) where the app asks the Android kernel to lookup socket ownership. Each query takes 1 to 2 milliseconds. If we process 5000 packets per second and query the kernel for each one, the CPU will be overloaded immediately.

**38. How do you prevent memory leaks (OOM) when `flowCache` stores thousands of connection keys in RAM over long periods?**
*   **Answer:** Yes, each connection entry in `flowCache` consumes only about 128 bytes. Even if a user opens 10,000 connections while browsing, it uses only 1.2 MB of RAM, which is tiny on modern devices. To be safe, I also plan to add an LRU Eviction policy to automatically clear connections that have been inactive for over 1 hour.

**39. Detail the failure counter logic (`failCountCache`) in `AppResolver.kt`. Why is it capped at 3 attempts?**
*   **Answer:** Yes, when a TCP connection is first opened (the SYN packet), the kernel takes a few milliseconds to write the socket to its internal table. If I only checked once and cached a failure (-1) immediately, I would miss the app's real UID. Testing up to 3 times (the first 3 packets) gives the kernel enough time to update, increasing accuracy.

**40. Why is Negative Caching crucial to protect the CPU?**
*   **Answer:** Yes, professors. Many network packets (like DNS, NTP, or system traffic) do not belong to a specific app, so the system call returns a failure (-1). Without negative caching, we would call the kernel for every single packet in these flows, overloading the CPU. Storing `-1` in the cache allows us to bypass these system flows instantly in RAM.

**41. How does "Retroactive Flow Migration" resolve the byte loss issue during the initial connection handshake?**
*   **Answer:** Yes, during the first 3 packets when the system is waiting to resolve the UID, their byte sizes are accumulated in a temporary virtual bucket mapped by port number. Once the UID is resolved at the 4th packet, all accumulated bytes are moved to the real app UID. This ensures telemetry is 100% accurate with zero byte loss.

**42. Why is `flowCache` in `DataPlaneProcessor.kt` declared as a `ConcurrentHashMap` rather than a standard `HashMap`?**
*   **Answer:** Yes, because packets are read from the TUN interface by multiple concurrent coroutines. If I used a standard `HashMap`, concurrent reads and writes from different threads would cause a `ConcurrentModificationException` and crash the app. `ConcurrentHashMap` ensures multi-thread safety.

**43. Why must `pm.getPackagesForUid(uid)` in `AppResolver.kt` be wrapped in a try-catch block?**
*   **Answer:** Yes, because if we query the package name for a special system UID that does not have an installed package, the Android system throws a `NameNotFoundException`. Without a `try-catch` block, this error would crash the entire QoS service.

**44. How does the system identify System Apps using bitwise flags on `ApplicationInfo`?**
*   **Answer:** Yes, I use a bitwise AND check: `(flags and ApplicationInfo.FLAG_SYSTEM) != 0`. If the result is non-zero, Android confirms it is a pre-installed system application, helping the system categorize it correctly.

**45. Explain the design of using the wildcard address `::` for IPv6 connections in `AppResolver.kt`.**
*   **Answer:** Yes, the device's IPv6 address changes constantly as the user moves. Therefore, instead of looking up a specific IP, I use the wildcard address `::` (which scans all local IPv6 addresses) combined with the port number to find the UID, avoiding lookup failures.

## CATEGORY V: BACKEND ARCHITECTURE, WASM SQLITE & TELEMETRY SYNC (Questions 46 - 55)

**46. Why did you choose WebAssembly SQLite (`sql.js`) over native SQLite drivers for the Node.js backend?**
*   **Answer:** Yes, native drivers require compiling C code for each operating system during installation, which often fails. `sql.js` runs entirely on WebAssembly on top of Node.js's JavaScript engine, ensuring it runs on Windows, Linux, and macOS without compilation issues.

**47. Analyze the data loss risk of the Throttled Background Saving mechanism (writing every 5 seconds) if the server crashes at second 4.**
*   **Answer:** Yes, if the server crashes at second 4, we will lose the last 4 seconds of telemetry data. However, since telemetry is only used to draw real-time graphs, losing 4 seconds of logs does not affect the actual bandwidth shaping. In return, this mechanism prevents the Node.js server from freezing due to constant disk writes.

**48. Why did you choose Node.js and Express for the Backend instead of Python (FastAPI/Flask) or Java Spring Boot?**
*   **Answer:** Yes, professors. I chose Node.js and Express because they are lightweight, fast to boot, and consume very little RAM on low-spec servers. Node.js uses a single-threaded, non-blocking Event Loop, which is perfect for handling high-frequency telemetry updates sent from multiple devices every 5 seconds without blocking the server. It is much more lightweight than Spring Boot, and handles real-time I/O concurrency more efficiently than Python.

**49. Draw a textual diagram of a packet's lifecycle: How does a packet travel from Chrome through your system to the Internet and back?**
*   **Answer:** Yes, professors. The packet journey consists of these steps:
    1. *Outbound:* Chrome initiates a connection -> Android OS redirects the packet into the virtual TUN interface.
    2. *Interception:* My app reads the raw bytes from the TUN -> forwards them to the `packetChannel` queue.
    3. *Analysis & Scheduling:* `DataPlaneProcessor` dequeues the packet -> invokes `AppResolver` to find Chrome's UID -> passes it to `BandwidthScheduler` to check the Token Bucket and calculate the WFQ priority.
    4. *Proxying:* If approved, `TcpRelayManager` performs the Split-Handshake with Chrome and uses a protected socket (`protect()`) to forward the data to the physical WAN.
    5. *Inbound:* The remote server replies to the proxy socket -> the proxy translates Sequence/Ack numbers -> writes the response packet back into the TUN -> Android delivers it to Chrome.

**50. The sync protocol uses a Batch Push model every 5 seconds. Why not sync telemetry immediately as packets are processed?**
*   **Answer:** Yes, sending telemetry for every single packet would require the phone to make thousands of HTTP requests per second. This would overheat the device and drain the battery in minutes. Batching requests every 5 seconds saves both battery and mobile data.

**51. At which layer of the OSI model does the virtual TUN interface operate? Why can you read IP headers but not MAC addresses?**
*   **Answer:** Yes, professors. The virtual TUN (Network Tunnel) interface operates at Layer 3 (Network Layer) of the OSI model. It processes raw IP packets. Unlike a TAP interface, which operates at Layer 2 (Data Link Layer) and processes Ethernet frames containing MAC addresses, the TUN interface designed by Android strips away the Ethernet header. Therefore, when my app reads bytes from the `/dev/tun` descriptor, the byte array starts directly with the IP Header, which is why we do not see MAC addresses.

**52. What does the `protect(socket)` call actually do at the OS level? How does it bypass the TUN interface routing?**
*   **Answer:** Yes, professors. Under the Linux kernel in Android, calling `protect(socket)` applies a system-level routing mark (Socket Mark) to that socket. The Android routing policy database (IP Rules) dictates that any packets originating from a socket with this mark must bypass the VPN routing table and go directly to the physical interface (Wi-Fi or cellular). This prevents the proxy socket's outgoing traffic from being routed back into the TUN interface, avoiding an infinite routing loop.

**53. How does the Node.js server protect its Wasm SQLite memory space from API data flood attacks (DDoS)?**
*   **Answer:** Yes, I set the maximum payload limit for the Express server to 1MB (`express.json({ limit: '1mb' })`). If an attacker sends massive payloads to flood the database, the server rejects them immediately, protecting the Wasm RAM.

**54. Why did you use Vanilla JavaScript and Chart.js for the Frontend Dashboard instead of popular frameworks like React, Vue, or Next.js?**
*   **Answer:** Yes, professors. The Web Admin dashboard is designed to show real-time telemetry charts and set simple policies. Using Vanilla JS and Chart.js makes the page load instantly, eliminates bundle size overhead, and avoids the complex build steps of frameworks like React or Vue. This keeps the frontend clean, responsive, and compatible with any browser without client-side dependencies.

**55. Why must you shape the total bandwidth target to be roughly 5% lower than the actual physical link speed? What if you set it equal to or higher?**
*   **Answer:** Yes, professors. This is critical for QoS enforcement. If the target bandwidth is set equal to or higher than the physical link capacity, the bottleneck remains at the external Wi-Fi router or cell tower. The physical router will fill its buffer first and drop or queue packets based on its own logic. This renders our mobile scheduler useless because the phone's queues will always be empty. Shaping the target 5% lower shifts the bottleneck to our virtual TUN interface, giving us full control to queue, prioritize, and drop packets according to our QoS policies.

*   **Answer:** Yes, standard deviation measures how much the speed fluctuates around the set limit. A low standard deviation (under 0.15 Mbps on a 5 Mbps limit) shows that the Token Bucket controls the speed stably without causing network jitter.

**74. How does the system handle changes in device battery power during long-term experimental testing?**
*   **Answer:** Yes, we monitored the current draw using Android's battery manager. Results show that when QoS is enabled, the device consumes about 8% less power during large downloads because TCP Backpressure reduces the active transmission cycles of the network radio.

**75. Did you perform experiments with multiple client devices connected to the hotspot simultaneously?**
*   **Answer:** Yes, we tested with 3 hotspot client devices downloading files at the same time. The WFQ scheduler divided the bandwidth equally among them with less than 5% speed difference, demonstrating fair allocation.

**76. Why do UDP throughput tests using iPerf3 show high Packet Loss Rates when QoS is enabled?**
*   **Answer:** Yes, since UDP does not automatically reduce its sending rate like TCP does, the client continues to push packets at full speed. To enforce the limit, the Token Bucket must drop the excess packets, resulting in a high packet loss rate at the receiver. This is the correct behavior of policing.

**77. How did you guarantee the objectivity and scientific validity of the experimental measurements in the thesis?**
*   **Answer:** Yes, to ensure objectivity, we repeated each experiment 50 times at different times of the day to eliminate random network noise. We then calculated the Mean, Standard Deviation, and 95% Confidence Intervals for the thesis data.

**78. If the committee asks: 'Why not deploy QoS at the Wi-Fi Router instead of building a complex system on the mobile device?', how will you defend your approach?**
*   **Answer:** Yes, professors. Mobile-side QoS offers three major advantages: First, it remains active when the user switches to 4G/5G or public Wi-Fi. Second, it identifies apps with 100% accuracy using OS UIDs, whereas routers are blind to encrypted HTTPS traffic (Port 443). Third, it allows users to customize priorities directly on their own devices.

**79. What are the current Technical Limitations of your QoS Scheduler, and how do you plan to address them in future work?**
*   **Answer:** Yes, there are two limits: First, it requires Android 10+ for UID resolution. We plan to research eBPF socket logging to support older Android versions. Second, routing packets through userspace adds a very small latency. We plan to rewrite the core packet processing logic in Rust or C++ using JNI to speed it up.

**80. If commercial applications (e.g., Facebook or YouTube) attempt to bypass your QoS limits by using aggressive application-layer (L7) mechanisms (such as opening multiple parallel TCP connections, streaming via QUIC/HTTP3 over UDP, or adaptive bitrate scaling), how does your system respond? Can they bypass the scheduler?**
*   **Answer:** Yes, professors. Large commercial applications **cannot bypass** our limits due to three design choices:
    1. *UID-Level Token Bucket:* We limit bandwidth by the app's UID (app ID) rather than per connection. Even if Facebook opens 10 or 100 connections, they all draw tokens from the same shared bucket, so the total speed remains capped.
    2. *QUIC/HTTP3 Interception:* For YouTube's QUIC protocol on UDP port 443, we intercept this port and apply the same Token Bucket rules, forcing QUIC to scale down its sending rate.
    3. *Adaptive Bitrate Synergy:* When bandwidth is throttled, the video player automatically lowers its resolution (e.g., from 1080p to 480p) to match the available speed, allowing smooth playback without buffering or hogging the network.

## CATEGORY VIII: THREAT MODELING & SYSTEM SECURITY (Questions 81 - 100)

**81. Can a hacker perform a Man-in-the-Middle (MitM) attack to decrypt HTTPS passwords through the virtual TUN interface?**
*   **Answer:** Yes, professors. Absolutely not. HTTPS is encrypted end-to-end between the phone and the server using SSL/TLS. Our proxy runs at Layer 4 (TCP), meaning it only relays the encrypted binary data and does not have the decryption keys. It is impossible for anyone to read passwords or private data through the proxy.

**82. Can a malicious app exploit the QoS Proxy to perform a 'Sandbox Escape' and steal bank credentials?**
*   **Answer:** Yes. Absolutely not. App isolation (Sandbox) is managed by the Linux kernel inside the Android OS. My proxy runs in userspace and only has access to network packets routed into the TUN interface. It cannot access the RAM or files of other apps, so it cannot bypass the Android Sandbox.

**83. If a malicious app floods the TUN with TCP SYN packets (local SYN Flood), how does `TcpRelayManager` prevent memory exhaustion?**
*   **Answer:** Yes, we have two layers of defense: First, Negative Caching quickly marks spam connections from unknown apps as `-1` in the cache, allowing the system to ignore them without using CPU. Second, we limit the socket connection backlog to 128 (`backlog = 128`). If it goes higher, Android drops the packets at the kernel level, protecting the RAM.

**84. How does the proxy protect users from Wi-Fi local network DNS spoofing?**
*   **Answer:** Yes, my app intercepts all DNS queries (Port 53) going into the TUN and forwards them directly to secure DNS servers (like Cloudflare `1.1.1.1` or Google `8.8.8.8`) that bypass the VPN. Spoofed DNS replies from the local Wi-Fi are ignored because they do not match the virtual sequence numbers managed by the proxy.

**85. Can an attacker exploit database update APIs to perform SQL Injection attacks on the Node.js backend?**
*   **Answer:** Yes. Absolutely not. The Node.js backend uses Parameterized Queries with the `sql.js` library. The telemetry data is passed as separate parameters and is never concatenated directly into SQL strings. SQLite treats them as plain strings, eliminating any SQL injection risks.

**86. What prevents a malicious app from spoofing a device ID (`device_id`) to post fake telemetry to the backend server?**
*   **Answer:** Yes, doing this would only mess up the telemetry graphs for that device ID on the Web Admin. It cannot modify the QoS policy on the actual device. The device pulls its configuration from the server using secure HTTPS connections, and policies are applied locally within the device's own TUN interface.

**87. How is the physical SQLite database file `qos_scheduler.db` secured from unauthorized local processes on the server?**
*   **Answer:** Yes, the database file `qos_scheduler.db` is stored inside the Node.js app folder and restricted at the OS level using `chmod 600`. This permission only allows the Node.js process to read and write the file, blocking other local processes.

**88. If a malicious app attempts to exploit a buffer overflow in `RawPacket.kt` by sending an IP packet with a malformed header length, how does your Kotlin code protect itself?**
*   **Answer:** Yes, in `RawPacket.kt`, before parsing a packet, I check the size of the byte array. If it is smaller than 20 bytes (minimum IP header size), the packet is ignored immediately. I also check the declared length against the actual packet size and discard mismatches, preventing buffer overflows.

**89. Why is the physical proxy socket connection not flagged as IP Spoofing by the cellular carrier?**
*   **Answer:** Yes, when the proxy connects to the WAN, it uses the real IP address assigned to the phone's Wi-Fi or LTE interface. The client's local IP is only simulated inside the virtual TUN interface. To the carrier, the traffic looks like standard, authorized data from the phone's real IP.

**90. On modern Android versions, SELinux blocks reading `/proc/net/arp`. How does `DeviceRegistry` resolve this to manage Hotspot QoS?**
*   **Answer:** Yes, on newer Android versions, SELinux blocks reading `/proc/net`. To resolve this, I use `ConnectivityManager.getLinkProperties()` to read Wi-Fi Hotspot routes and listen to tethering broadcasts (`ACTION_TETHER_STATE_CHANGED`) to get client IPs and MACs without reading system files.

**91. How does the proxy prevent memory leaks when connections are terminated abruptly (e.g., toggle Airplane Mode)?**
*   **Answer:** Yes, when Airplane Mode is turned on, Android shuts down interfaces and fires a network loss event. My app listens to this event, immediately closes all active `SocketChannels`, and clears write queues. Using `try-finally` blocks ensures that resources are always released safely during sudden network losses.

**92. How do you protect the proxy's UDP sockets from DNS Amplification attacks?**
*   **Answer:** Yes, DNS Amplification requires spoofing the victim's source IP. Since our proxy sockets are bound to the phone's physical interface, the Android kernel automatically enforces the device's real IP as the source. Attackers cannot spoof the source IP through our proxy, blocking this attack.

**93. What prevents a malicious app from injecting fake TCP `FIN` or `RST` packets into the TUN interface to terminate other apps' connections?**
*   **Answer:** Yes, Android isolates apps using Sandbox UIDs. A standard app has no permissions to write to the File Descriptor of the virtual TUN interface (`/dev/tun`), which belongs exclusively to my `VpnService` process. Thus, malicious apps cannot inject fake packets.

**94. Why did you use HTTP REST instead of HTTPS to sync telemetry in the testing environment? What are the real-world risks, and how would you fix them?**
*   **Answer:** Yes, in the local test environment, using HTTP made it easy to test without dealing with self-signed SSL certificates on the phone. In the real world, this could allow someone on the same Wi-Fi to sniff app names. To secure this, we must upgrade to HTTPS using certificates (like Let's Encrypt) and use JWT token authorization in headers.

**95. How do you protect the policy update API (`POST /api/policies`) on the Node.js server from unauthorized LAN clients?**
*   **Answer:** Yes, I use a shared secret (API Key) on both the server and the app. When updating a policy, the client must send this key in the `X-API-KEY` header. The server matches the key before modifying the database, blocking unauthorized LAN clients.

**96. Why does your proxy not support HTTP URL Filtering?**
*   **Answer:** Yes, because almost all web traffic is encrypted using HTTPS. The data going through the proxy is encrypted binary data, so parsing it for URLs is impossible and wastes CPU. Therefore, we categorize traffic by destination port and IP address, which is fast and effective.

**97. What happens if an attacker sends an IP packet with a Total Length field larger than the actual received buffer?**
*   **Answer:** Yes, in `RawPacket.kt`, I do not trust the length field in the IP header. I use the actual read size from the network card. All array slicing uses `minOf(totalLength, length)` as a boundary, preventing Out-of-Bounds memory errors.

**98. How do you protect the Wasm SQLite database memory space if a client generates millions of telemetry entries (DDoS telemetry)?**
*   **Answer:** Yes, I set a maximum limit of 50,000 rows for the telemetry table in RAM. If the row count exceeds this limit, the server automatically triggers an Emergency Pruning to delete the oldest 20% of the entries immediately, keeping RAM safe.

**99. Why does `Ipv4TcpPacketCodec.kt` use bitwise shifts (`and 0xFF`, `shr`, `shl`) to write checksums instead of standard Java conversion libraries?**
*   **Answer:** Yes, bitwise operations (`and 0xFF`, `shr`, `shl`) run directly on CPU registers in a single clock cycle. Standard Java libraries create temporary objects in RAM, which slows down packet processing and consumes more battery.

**100. How does your thesis resolve the conflict between application-level QoS and Android's Sandbox security?**
*   **Answer:** Yes, professors. My thesis resolves this conflict by implementing a userspace Split-Handshake Proxy that works with the official `VpnService` API. This allows 100% app identification using kernel UIDs and precise bandwidth capping without requiring root access, keeping Android's Sandbox security intact while achieving high throughput via cache-optimized Binder IPCs.
